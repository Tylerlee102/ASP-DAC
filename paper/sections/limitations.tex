\section{Limitations}

The scope is single-hart RV32I firmware with deterministic core semantics, the same program image, the same initial architectural state, and modeled external events. The current theorem excludes multicore interleavings, DMA unless modeled, cache/coherence effects unless modeled, analog nondeterminism, and claims about undefined C behavior.

MMIO side effects are modeled only at the replay-visible boundary. Interrupts are injected at an architectural boundary. Replay consumes every required event exactly once; missing, duplicate, reordered, or corrupted events must fail comparison. Capsule overflow invalidates replay availability for failure reconstruction unless the replay-critical prefix is still complete.

The replay-consume path is host-driven in the Verilator harness. The mapped design measures the recorder/instrumented full-core wrapper, not a synthesizable autonomous replay-consume engine. The v1 workload evidence recovers 375/375 replay PASS rows only by using workload-aware depths: 256 for smoke and short, 1024 for medium, 4096 for long, and 16384 for stress. These depths increase capsule bytes and reduce the median stress-workload reduction versus a raw full-instruction trace to 22.84\%. High FF/LUT overhead remains an optimization target: after gating unselected v2 recorders in v1 mapped builds, same-target ECP5 overhead is still as high as 221.23\% LUT and 646.43\% FF. ASIC power/area/timing is not evaluated, and no generic-cell or open-PDK result should be inferred from the FPGA flow. The evaluation is single-hart RV32I and does not cover multicore, DMA, caches/coherence, or arbitrary peripherals.

ReplayCapsule-RV v2 reduces the architectural risk by adding compressed recorder RTL and a synthesizable replay-consume controller prototype, but the prototype is not fully integrated into the full-core replay harness. The consumer has tested stream consumption, order checking, MMIO/interrupt checking, and payload-hash rejection, and it maps as a standalone ECP5 block. It is still missing the full-core replay datapath: capsule storage/streaming in the full-core top, core MMIO/IRQ replay muxing, observed-event plumbing from PicoRV32, and an autonomous replay-mode controller. These rows should not be used to claim autonomous full-core hardware replay.
