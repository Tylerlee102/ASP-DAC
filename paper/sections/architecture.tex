\section{Hardware Architecture}

ReplayCapsule-RV instruments the processor boundary rather than replacing the core. The PicoRV32 wrapper exposes memory, MMIO, interrupt, and commit-context signals to the record-side modules. The event tap observes candidate facts, the classifier selects replay-relevant and diagnostic classes, the slicer retains context around a property failure, the capsule buffer stores encoded packets, and the property checker freezes the capsule on a target violation.

The board-level mapped wrappers expose only clock, reset, UART, GPIO, status LEDs, and three recorder-status pins. Large debug and internal buses remain inside the FPGA design. A v1 mapped build disables redundant v2 recorder instances that are not selected by the architecture parameter, but the remaining recorder, event packet storage, signature path, property checker, and status visibility still cost substantial resources. The current same-target ECP5 rows report LUT overhead from 114.20\% to 221.23\%, FF overhead from 174.53\% to 646.43\%, and Fmax change from -19.56\% to -9.98\%. This overhead is acceptable for the intended bring-up/debug-instrumentation use case, where fidelity and auditability matter more than shipped-silicon area; it is not presented as an optimized always-on production block.

The selected v2 minimal recorder profile is a separate compile-time mapped path. It removes the property checker, context slicer, on-chip event buffer, hash signature, and detailed counters from the recorder instance, while still emitting commit-indexed MMIO read/write, external-input, and interrupt enter/exit events for host capture. On the same ECP5 point, that profile maps at 2818 LUT, 992 FF, 3 BRAM, and 68.82 MHz against a 2603 LUT, 956 FF, 3 BRAM, and 68.85 MHz baseline. The core and hashed v2 rows remain diagnostic-rich comparison configurations.

\subsection{ReplayCapsule-RV v2 Prototype}

ReplayCapsule-RV v2 keeps the v1 recorder intact and adds separately selectable compressed recorder profiles. The diagnostic v2 path packs replay-critical event class, commit-delta, address-token, payload-hash, and limited debug context into 64-bit capsule words; repeated MMIO addresses can use a small dictionary, and diagnostic context can be dropped under adaptive buffer pressure while replay-critical MMIO, interrupt, property-fail, and checkpoint-hash events remain capturable. The selected minimal recorder profile keeps only the replay-driving boundary-event stream. The v2 implementation also includes a synthesizable replay-consume controller prototype that validates event type, commit order, MMIO value, interrupt cause, and payload hash before reporting that all capsule events were consumed.

The v2 recorder and replay consumer are not presented as a standalone board/silicon replay engine. In this artifact, v2 recorder RTL is selectable in the full-core record path, the RTL source has a capture-store path that retains replay-critical words across reset in directed HDL tests and v2 record signatures, and a replay-mode controller now clears/arms that store and starts captured-store replay from the retained count. The replay-consume controller has directed RTL corruption tests, saved-capsule full-core checks, and same-instance self-replay checks in which the replay run streams from the captured RTL store without harness-preloaded capsule words. During replay, that source streams the capsule while the consumer validates observed boundary events and drives core-facing MMIO read data and IRQ replay through the PicoRV32 wrapper. The remaining missing piece is a standalone board/silicon replay flow outside the Verilator reset orchestration and memory/peripheral shell.
