# ReplayCapsule-RV Related Work Matrix

Status date: 2026-06-25

Scope assumption: ReplayCapsule-RV is a serious hardware/architecture research prototype for event-sufficient deterministic replay of embedded RV32I failures under interrupts and MMIO. This matrix is aligned with the local architecture, event model, capsule format, research plan, and replay sufficiency theorem docs present in `docs/` as of this update. It should still be tightened once RTL, firmware benchmarks, and evaluation results exist.

Legend:
- Verified: checked against an official standard, publisher page, or publicly available paper.
- Partially verified: title/venue/abstract or high-level description checked, but detailed mechanisms still need the full paper.
- Needs verification: plausible category-level comparison, but do not cite as fact until a primary source is read.

## Local ReplayCapsule-RV Contract Used for Comparison

The local docs define ReplayCapsule-RV as:

- single-hart RV32I embedded replay in v1;
- no full instruction trace and no hidden device-state reconstruction;
- commit-index/`retire_count` event anchoring, with optional `cycle_delta` only when wait-state behavior is replay-visible;
- capsule events including `SYNC`, `IRQ_LEVEL`, optional checker-only `IRQ_TAKEN`, `MMIO_READ`, `MMIO_WRITE`, slice/hash/error/end records;
- MMIO reads as replay inputs and MMIO writes as replay-validated observations;
- deterministic replay only under fixed firmware image, platform profile, initial state/reset inputs, and exact boundary events;
- explicit exclusions for multicore races, DMA or external memory writers unless separately modeled, arbitrary peripheral modeling, analog timing, and undefined behavior.

## Cross-Cutting Risk Notes

1. Do not call ReplayCapsule-RV "first deterministic replay," "first hardware replay," "first embedded black box," or "first RISC-V trace" without a systematic survey.
2. Do not claim full determinism for all devices. Use "event-sufficient for the scoped RV32I platform and failure class" until there is a proof or exhaustive evaluation.
3. Do not compare compression ratio, area, timing, power, or replay success rate against E-Trace, N-Trace, ENCORE, Hassert, REMU, Vidi, or post-silicon trace compression until measured with matched baselines.
4. Do not imply RISC-V standard compliance unless the prototype implements the relevant Debug, E-Trace, N-Trace, or Trace Control Interface behavior and passes a conformance check.
5. Treat "MMIO replay" carefully. Recorded MMIO values may be path-dependent on interrupt timing and firmware writes; source work such as Conware explicitly warns that naive MMIO-value replay can fail across behavior changes.
