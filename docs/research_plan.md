# ReplayCapsule-RV Research Plan

## Objective

Build a serious, reviewable FPGA/RTL research prototype for deterministic replay of embedded RV32I safety failures triggered by interrupts and memory-mapped I/O.

The central question is:

> What is the minimum hardware event set needed to deterministically reproduce a safety failure in a tiny embedded RISC-V system with interrupts and MMIO?

## Precise Contribution

ReplayCapsule-RV should contribute:

1. A formal event-sufficiency model for single-core embedded RV32I systems.
2. A replay sufficiency theorem for failures caused by modeled nondeterministic boundary events.
3. Synthesizable hardware that captures property-relevant events and freezes a replay capsule at failure.
4. A deterministic replay testbench with cycle-index and commit-index modes.
5. A bug benchmark suite with interrupts, MMIO, and runtime safety properties.
6. Baselines and ablations that show which event types matter.
7. Real simulation and synthesis results only.

## Current Environment Finding

The workspace started empty and was not a git repository. Local `verilator`, `yosys`, and `make` commands were not available at kickoff, so Phase 1/2 verification began with Python smoke tests and static RTL checks. Current runs can use workspace-local `yowasp-yosys` for generic synthesis and OSS CAD Suite for Icarus directed simulations, four PicoRV32 wrapper firmware smokes, and Verilator lint-only checks. `make`, a C++ compiler path, and the RISC-V compiler remain unavailable locally, so full benchmark-wide firmware-running RTL simulation and mapped hardware metrics remain TODO/NA until those tools are installed or supplied by CI.

## Phase 0 Gate

Deliverables:

- Repository structure.
- `docs/research_plan.md`.
- Subagent ownership plans under `docs/subagents/`.
- Initial honesty policy: no fake measurements, no fake novelty, no unsupported "first ever" language.

## Phase 1 Gate

Deliverables:

- Minimal RV32I-style embedded system scaffold.
- Instruction/data memory boundary.
- Timer interrupt source.
- MMIO sensor input.
- MMIO actuator/PWM output.
- UART or command register.
- Deterministic seeded input mode.
- Randomized interrupt mode plan.
- Hello-world and MMIO smoke firmware sources.

Initial implementation uses a deterministic Python event model and an RTL SoC scaffold while PicoRV32/Verilator integration is pending local tool availability.

## Phase 2 Gate

Deliverables:

- Event tap and event stream.
- Basic commit, branch, jump, store, load, MMIO, interrupt, external-input, property-fail, and checkpoint-hash event types.
- Tests for event ordering, MMIO value capture, interrupt timing, and replay comparison.

## Honesty Rules

- Do not report synthesis numbers until generated by a real synthesis flow.
- Do not report replay success until the replay scripts observe matching failures.
- Do not claim minimality globally; claim sufficiency under the defined model and use ablations to support practical minimal event sets per benchmark.
- Do not claim multicore, cache coherence, DMA, analog nondeterminism, undefined C behavior, or arbitrary external bus reordering support unless explicitly modeled and tested.
